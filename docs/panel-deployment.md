# Panel Deployment

The fleet panel is optional. Agents continue to monitor locally and push signed, bounded telemetry only when `[panel].enabled = true`.

## Deployment Modes

| Mode | Best for | Storage | Realtime |
| --- | --- | --- | --- |
| Cloudflare Worker/D1 | Low-maintenance public HTTPS panel without running another VPS service. | Cloudflare D1 | REST incremental loading. Durable Objects are required for true cross-request WebSocket broadcast. |
| Self-hosted Rust panel | Full backend control, private deployment, and WebSocket refresh. | SQLite, PostgreSQL, or MySQL | WebSocket refresh events with dataset scope. |

## Tokens and Paths

| Setting | Meaning | Required |
| --- | --- | --- |
| `PANEL_SHARED_SECRET` | HMAC ingest secret. Agents must use the same value in `[panel].secret`. | Required unless `PANEL_NODE_SECRETS` is used. |
| `PANEL_NODE_SECRETS` | Optional JSON map of per-node ingest secrets keyed by non-sensitive `panel.node_name`. | Optional. |
| `PANEL_OPERATOR_TOKEN` | Browser token for redacted operator pages. | Generated by deployment scripts when omitted. |
| `PANEL_ADMIN_TOKEN` | Browser token for management actions such as reviews. | Generated by deployment scripts when omitted. |
| `PANEL_VIEW_TOKEN` | Legacy alias for operator access. | Optional compatibility only. |
| `PANEL_ADMIN_PATH` | Browser management entry path. It is not the API security boundary; tokens still protect APIs. | Generated by deployment scripts when omitted. |
| `PANEL_PUBLIC_PAGES` | Pages visible without a token, for example `overview,probe_sources,nodes`. | Optional. |

New deployments should not use a fixed `/admin` path. The deployment scripts generate a random path like `/4f9a12d0c8ab` and save it with the generated tokens.

## Cloudflare Worker/D1

1. Build the static UI:

```bash
cd panel/ui
npm install
npm run build:web
cd ../..
```

2. Deploy with Wrangler:

```bash
CLOUDFLARE_ACCOUNT_ID="<account-id>" \
scripts/deploy-cloudflare-panel.sh
```

For CI or a non-interactive VPS, set `CLOUDFLARE_API_TOKEN` in the shell. Do not commit Cloudflare credentials.

The script creates or reuses D1, applies compatible schema migrations, deploys the Worker and `panel/web`, writes Worker secrets, and verifies `GET /api/v1/settings`.

Generated panel credentials are saved locally:

```bash
cat ~/.config/vps-sentinel/cloudflare-panel.env
```

Use this file to get:

- `PANEL_SHARED_SECRET`: put this into every agent as `[panel].secret`;
- `PANEL_OPERATOR_TOKEN`: use this for operator access;
- `PANEL_ADMIN_TOKEN`: use this on the management path;
- `PANEL_ADMIN_PATH`: open this path in the browser, for example `https://worker.example.workers.dev/<random-path>`.

You can override the credential file path:

```bash
PANEL_CREDENTIAL_FILE=/secure/path/panel.env scripts/deploy-cloudflare-panel.sh
```

To keep existing agents working during redeploy, reuse the saved credential file or explicitly export the same `PANEL_SHARED_SECRET`.

3. Configure agents:

```toml
[panel]
enabled = true
url = "https://vps-sentinel-panel.<your-workers-subdomain>.workers.dev/api/v1/ingest"
node_name = "prod-sg-1"
secret = "<PANEL_SHARED_SECRET from cloudflare-panel.env>"
privacy_mode = "strict"
```

4. Verify:

```bash
sudo vs panel push
curl -fsS https://vps-sentinel-panel.<your-workers-subdomain>.workers.dev/api/v1/settings
```

Cloudflare Worker mode currently returns `stream_unavailable` for `/api/v1/stream-ticket`. A plain Worker can handle API requests and D1 storage, but reliable browser broadcast after agent ingest requires Durable Objects or another stateful fan-out service. The UI still updates data without full-page reloads through scoped REST loading.

## Self-Hosted Rust Panel

1. Build and install:

```bash
cargo build --release --bin vps-sentinel-panel
sudo install -m 0755 target/release/vps-sentinel-panel /usr/local/bin/vps-sentinel-panel
sudo mkdir -p /usr/local/share/vps-sentinel/panel
sudo cp -a panel/web /usr/local/share/vps-sentinel/panel/web
```

2. Generate a secure environment file:

```bash
sudo PANEL_ENV_FILE=/etc/vps-sentinel-panel/panel.env scripts/create-panel-env.sh
sudo cat /etc/vps-sentinel-panel/panel.env
```

The generator preserves existing values when the file already exists. It fills in random `PANEL_SHARED_SECRET`, `PANEL_OPERATOR_TOKEN`, `PANEL_ADMIN_TOKEN`, and `PANEL_ADMIN_PATH` when they are missing.

Random values use `openssl` when available, then `/dev/urandom` with `od`, then Node.js as a fallback. On very small systems, install `openssl` or set the four `PANEL_*` values explicitly before running the script.

Optional overrides:

```bash
sudo PANEL_DB_BACKEND=postgres \
PANEL_DATABASE_URL='postgres://vps_sentinel:password@127.0.0.1:5432/vps_sentinel' \
scripts/create-panel-env.sh
```

3. Create the systemd service:

```bash
sudo useradd --system --home /var/lib/vps-sentinel-panel --shell /usr/sbin/nologin vps-sentinel-panel || true
sudo install -d -m 0750 -o vps-sentinel-panel -g vps-sentinel-panel /var/lib/vps-sentinel-panel
sudo tee /etc/systemd/system/vps-sentinel-panel.service >/dev/null <<'EOF'
[Unit]
Description=vps-sentinel fleet panel
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=vps-sentinel-panel
Group=vps-sentinel-panel
EnvironmentFile=/etc/vps-sentinel-panel/panel.env
ExecStart=/usr/local/bin/vps-sentinel-panel
Restart=on-failure
RestartSec=5s
NoNewPrivileges=true
ProtectSystem=full
ProtectHome=true
ReadWritePaths=/var/lib/vps-sentinel-panel

[Install]
WantedBy=multi-user.target
EOF
sudo systemctl daemon-reload
sudo systemctl enable --now vps-sentinel-panel
```

4. Put the service behind HTTPS using Nginx, Caddy, or another reverse proxy. Set agent `[panel].url` to `https://your-panel-domain/api/v1/ingest`.

## Storage Backends

SQLite is the default:

```bash
PANEL_DB_BACKEND=sqlite
PANEL_DATABASE_URL=sqlite:///var/lib/vps-sentinel-panel/panel.db
```

PostgreSQL:

```bash
PANEL_DB_BACKEND=postgres
PANEL_DATABASE_URL=postgres://vps_sentinel:password@127.0.0.1:5432/vps_sentinel
```

MySQL:

```bash
PANEL_DB_BACKEND=mysql
PANEL_DATABASE_URL=mysql://vps_sentinel:password@127.0.0.1:3306/vps_sentinel
```

The Rust panel initializes compatible schema on startup.

## Themes

Themes are registered with `PANEL_THEMES`, for example:

```bash
PANEL_THEMES='default:Default,ocean:Ocean'
PANEL_THEME='default'
```

Theme files live under `panel/web/themes/<theme-id>/` and should only contain static CSS/JSON assets.
